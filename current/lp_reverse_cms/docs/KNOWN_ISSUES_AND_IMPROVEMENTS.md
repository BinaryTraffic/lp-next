## 1. アーキテクチャ上の問題

### 1-1. 🔴 JSON ファイルストレージの並行書き込みリスク

**問題:**  
`lp_structure.json` / `client_data.json` / `asset_map.json` などすべてのデータが JSON ファイルに保存されている。複数タブや複数リクエストが同時に書き込んだ場合、ファイルが破損する可能性がある。現状はロック機構（`flock`）が一部にしかない。

**影響:** データ消失・JSON parse エラー・解析結果の上書き  

**改善案:**
- 全書き込み箇所に `flock(LOCK_EX)` を徹底する
- 将来的には SQLite に移行することで並行安全性を確保（Composer 不要・DBサーバー不要を維持できる）
- ワークスペース単位でのファイルロックマネージャクラスを作る

---

### 1-2. 🟡 バックグラウンドジョブが PHP CLI プロセス依存

**問題:**  
解析・生成のバックグラウンドジョブは `proc_open()` または `exec()` で PHP CLI プロセスを起動している。Apache/PHP の設定によっては `exec` が無効化されていることがあり、この場合ジョブが起動しない。またプロセスが孤児になったときのクリーンアップが不完全。

**影響:** ホスティング環境（共有サーバー等）での動作不可、ゾンビプロセス蓄積  

**改善案:**
- ジョブキューを JSON ファイルベース（`jobs/queue.json`）で実装し、Cron（毎分）でワーカーを起動するアーキテクチャに移行
- `exec` 不可環境での graceful フォールバック（同期実行モード）を提供

---

### 1-3. 🟡 ワークスペースの GC（ガベージコレクション）未整備

**問題:**  
`data/ws_xxx/` と `output/ws_xxx/` はセッション失効後も残り続ける。長期運用でディスクが圧迫される。`workspace_delete.php` は手動削除のみ。

**改善案:**
- 最終アクセスから N 日後に自動削除する Cron ジョブ
- ダッシュボードにワークスペース容量一覧と一括削除機能（管理者向け）

---

### 1-4. 🟢 マルチテナント対応の欠如

**問題:**  
現状、全ユーザーが同一サーバーの同一 `data/` / `output/` を共有している。ユーザー A のワークスペース ID を知っていればアクセスできてしまう（セキュリティより利便性を優先したMVP設計）。

**改善案:**
- ワークスペース ID にユーザーメールを紐付け、アクセス時にセッションユーザーと一致確認
- または `data/ws_xxx/owner.txt` で所有者チェック

## 2. 解析フェーズの問題

### 2-1. 🔴 JavaScript レンダリングコンテンツの取得不可

**問題:**  
`LpFetcher` は `cURL` でプレーン HTML を取得するため、**JS で動的生成されるコンテンツ（React / Vue / Next.js SPA等）は取得できない**。解析後の構造 JSON が空に近くなる。

**影響:** モダンな JS-heavy なサイトはクローン精度が著しく低い

**改善案:**
- Puppeteer/Playwright（Node.js）または Splash（Python）をサイドカーとして起動し、JS レンダリング後の HTML を受け取る API を作る
- またはフロントの「URLを解析」ステップで「静的取得 / JS レンダリング取得」を選択できるようにする

---

### 2-2. 🟡 CSS @media 内の `background-image: url()` の見逃し

**問題:**  
`LpAnalyzer` / `LpAssetDownloader` は CSS の `url()` を収集するが、`@media` クエリの中に入っている `background-image` が収集から漏れるケースがある。特にスマートフォン向け背景画像（`@media (max-width: 768px)` 内）が未取得になる。

**改善案:**
- CSS パーサーで `@media` ブロックを再帰的にスキャンする
- `preg_match_all` のパターンを `@media` を無視しないものに修正

---

### 2-3. 🟡 解析タイムアウト設定の分散

**問題:**  
解析ワーカーのタイムアウト値が `php.ini`・`analyze_worker.php` 内のハードコード・Apache `Timeout` ディレクティブに分散している。大規模サイトでどこかがタイムアウトしても、エラーメッセージが不明確。

**改善案:**
- `.env` に `LP_ANALYZE_TIMEOUT_SEC`（例: 300）を一元管理
- タイムアウト時のエラーメッセージを「どの処理でタイムアウトしたか」まで表示

---

### 2-4. 🟢 `<noscript>` 内コンテンツの扱い

**問題:**  
`<noscript>` 内の画像・テキストは解析対象外になっている場合がある。JS 無効時のフォールバック画像が含まれないことがある。

**改善案:**
- `<noscript>` 内の `<img>` も `img_hints` に追加する

---

## 3. アセット取得の問題

### 3-1. 🔴 CORS / Bot ブロック対策なし

**問題:**  
一部のサイトは cURL User-Agent を Bot として検出し `403` / `429` を返す。`fetch_failures.json` に記録されるが、ユーザーへの案内が不十分。

**影響:** 重要な画像・CSS が取得できず、生成 HTML が崩れる

**改善案:**
- User-Agent を実際のブラウザ文字列に変更（現在は `LpFetcher/1.x`）
- `Referer` ヘッダーを元の LP URL に設定
- 取得失敗時に「この URL は Bot ブロックの可能性があります」と UI でアドバイス

---

### 3-2. 🟡 fetch_failures.json のフォーマット変遷

**問題:**  
v1.5.006 で `fetch_failures.json` の形式が `["url1","url2"]`（旧）から `[{url, http_code, reason}]`（新）に変更された。互換レイヤーは `debug.php` にのみ実装されており、他のファイルで旧フォーマットを期待するコードが残っている可能性がある。

**改善案:**
- `LpAssetAudit` 等、`fetch_failures.json` を読む全箇所で正規化処理を一元化する
- `LpWorkspace::readFetchFailureUrls()` のような共通メソッドを作る

---

### 3-3. 🟡 画像メタデータ（w/h/mime）の解析時記録が未完了

**問題:**  
`LpAssetDownloader` で `getimagesize()` を使い `rollback_w`/`rollback_h`/`rollback_mime` を `lp_structure.json` に保存するよう v1.5.x で追加したが、**CSS `background-image` 由来の画像**には適用されておらず、`img[src]` 由来の画像のみに限定されている。

**影響:** 背景画像の寸法情報が JSON に入らないため、ブレンド処理の精度に影響

**改善案:**
- `css_background_hints` 経由でダウンロードされた画像にも `getimagesize()` を適用
- `LpAssetDownloader::downloadUrl()` の共通化でパスの違いに関わらず一律取得

---

### 3-4. 🟢 フォント（woff2）の取得失敗が多い

**問題:**  
Google Fonts は直接の cURL では CSS を返すが、その CSS から派生する `.woff2` の取得で `403` になるケースがある。Adobe Fonts（Typekit）は認証が必要で取得不可。

**改善案:**
- フォントのみ「外部参照のまま残す」オプションを提供
- Google Fonts の URL を自動検出し、`@font-face` 宣言を `font.googleapis.com` からセルフホスト用に変換するヘルパーを追加

---

## 4. 生成フェーズの問題

### 4-1. 🔴 `{%代替URL%}` プレースホルダーの URL エンコード問題

**問題:**  
ブラウザは `href="{%代替URL%}/page"` を `href="%7B%25%E4%BB%A3%E6%9B%BF..." ` のようにエンコードする。プレビューのクリックガードでは `host.toLowerCase()` に `%7b` や `{` が含まれることを検出しているが、ブラウザのバージョンや文字コードの違いでエンコード形式が変わる可能性がある。

**改善案:**
- `{%代替URL%}` のような日本語プレースホルダーをやめ、ASCII のみの識別子（例: `__LP_PLACEHOLDER__`）に変更
- プレビュー側の検出ロジックをより堅牢にする（`decodeURIComponent` で正規化してから比較）

---

### 4-2. 🟡 生成 HTML の外部参照残存（output_unreplaced）

**問題:**  
`output_unreplaced.json` に記録される「置換漏れ外部 URL」がゼロにならないサイトが存在する。特に動的に生成された `<script>` 内の URL・インライン `style` 属性の `url()` が残りやすい。

**改善案:**
- `<script>` 内の文字列リテラルに含まれる URL も `asset_map` で変換を試みる
- インライン `style` 属性の `url(...)` を `LpGenerator::applyAssetMap` で確実に処理

---

### 4-3. 🟡 内部ページ（multipage）生成の品質

**問題:**  
内部ページ（`store/analyze_internal_page.php`）の解析は別フェーズとして実装されているが、内部ページ固有のアセット（トップページに無い CSS・画像）が `asset_map` に入らないことがある。

**改善案:**
- 内部ページ解析時に `LpAssetDownloader` を呼び直し、`asset_map` をマージする
- 内部ページの `lp_structure` を統合管理する専用クラスの整備

---

## 5. プレビューの問題

### 5-1. 🟡 ナビゲーションオーバーレイの信頼性

**問題:**  
`preview.php` のページ遷移インジケーター（`#navLoadingOverlay`）は `postMessage({type:"lp-nav-start"})` をトリガーに表示し、iframe の `load` イベントで非表示にする。ナビゲーションが失敗した場合（存在しないページへの遷移）は `load` が発火しないため、オーバーレイが残り続けることがある。現在は `pointer-events: none` で回避しているが、スピナーが出っぱなしになる。

**改善案:**
- `load` イベントだけでなく、`loadstart` / `error` でも制御する
- フォールバックタイムアウトを短くするだけでなく、失敗理由をユーザーに通知する

---

### 5-2. 🟢 iframe セキュリティポリシー

**問題:**  
iframe で読み込まれた生成 HTML は `allow-scripts same-origin` の sandbox なし。生成 HTML に残存した JS がプレビュー画面の親フレームに影響する可能性がある（CSP 未設定）。

**改善案:**
- `<iframe sandbox="allow-scripts allow-same-origin allow-forms">` を設定
- Apache で `Content-Security-Policy` ヘッダーを追加

---

## 6. UI / UX の問題

### 6-1. 🟡 サイドバーの高さ計算が固定値依存

**問題:**  
`#step2ActionBar` の `top` / `height` は `--lp-nav-h` CSS カスタムプロパティ（JS で実測）に依存している。ナビバー以外に固定ヘッダーが追加された場合や、ブラウザズームで高さが変わった場合に追従しない可能性がある。

**改善案:**
- `ResizeObserver` で navbar の高さを監視し、変化時に `--lp-nav-h` を更新

---

### 6-2. 🟡 モバイル / タブレット未対応

**問題:**  
管理 UI（`index.php`）は幅 1000px 以上を前提とした設計で、スマートフォン・タブレットでは `#step2Explorer` の二列レイアウトが崩れる。

**改善案:**
- `#step2Explorer` をスモール画面では縦積みに変更（`flex-direction: column`）
- `#step2ActionBar` をボトムナビバーとして配置する小画面レイアウト

---

### 6-3. 🟡 編集のアンドゥ（Undo）機能なし

**問題:**  
テキスト入力・画像差し替えは即時 `client_data.json` に保存されず、「保存＆サイト生成」を押すまで確定しない。しかし画像ブレンドはキャンバス操作のため、元に戻す手段が `rollback` ボタン（オリジナル画像に戻す）しかない。

**改善案:**
- 画像操作の履歴スタックをセッションストレージに保持し、ブラウザバック感覚で戻せるようにする
- 少なくとも「直前の状態に戻す」ボタンを各画像フォームに追加

---

### 6-4. 🟢 診断（debug.php）の UI 化

**問題:**  
`store/debug.php` は JSON をそのまま返すため、確認には DevTools または別ツールが必要。非エンジニアのユーザーが問題を把握しにくい。

**改善案:**
- Step 3（サイト確認）タブに診断サマリーを見やすい形で表示（既存の `step3DiagSummary` を拡充）
- 「未取得アセット一覧」「外部 URL 残存一覧」をクリック可能なテーブルで表示

---

### 6-5. 🟢 複数ページ（内部ページ）の編集 UI が弱い

**問題:**  
内部ページの編集は `#pageTree` からページを選択して行うが、ページ間の移動がワンクリックではなくツリー操作が必要。また全ページの一括テキスト検索・置換がない。

**改善案:**
- サイト全体のテキスト検索（`Ctrl+Shift+F` 相当）機能
- ページをまたいだ会社名・電話番号の一括置換

---

## 7. セキュリティ上の懸念

### 7-1. 🔴 SSRF（Server-Side Request Forgery）リスク

**問題:**  
`fetch_lp.php` / `analyze_lp.php` はユーザー入力の URL に対して cURL リクエストを送信する。悪意あるユーザーが `http://169.254.169.254/` （AWS/GCP メタデータエンドポイント）などを指定した場合、内部ネットワークの情報が漏洩する可能性がある。

**現状:** Google OAuth によるログイン必須のため外部からの悪用リスクは低いが、内部ユーザーの悪用は防げない。

**改善案:**
```php
// LpFetcher で IP アドレスのバリデーション追加
function isAllowedUrl(string $url): bool {
    $host = parse_url($url, PHP_URL_HOST);
    $ip = gethostbyname($host);
    // プライベート IP / ループバック / リンクローカル を拒否
    return !filter_var($ip, FILTER_VALIDATE_IP, 
        FILTER_FLAG_NO_PRIV_RANGE | FILTER_FLAG_NO_RES_RANGE
    );
}
```

---

### 7-2. 🟡 CSRF トークンの未適用エンドポイントが存在する可能性

**問題:**  
`lp_reverse_csrf.php` で CSRF トークン検証の仕組みは存在するが、全 `store/*.php` エンドポイントで一律に適用されているか確認が取れていない。特に画像アップロード・テキスト保存系のエンドポイントでの検証漏れリスク。

**改善案:**
- 全 `store/*.php` の冒頭で CSRF トークン検証を行うミドルウェア的な関数を必須化
- 単体テスト or grep で検証漏れを自動検出する CI を整備

---

### 7-3. 🟡 生成 HTML 内の XSS リスク

**問題:**  
元の LP のテキスト・属性値を `client_data.json` → 生成 HTML に出力する際、ユーザーが `<script>` タグを含む文字列を入力した場合に XSS が発生する可能性がある。プレビュー iframe 上では影響が出る。

**改善案:**
- `client_data.json` に保存する前に HTML タグをエスケープ or ストリップ
- または `Content-Security-Policy` でインライン JS を禁止するヘッダーを生成 HTML に付与

---

### 7-4. 🟢 ファイルアップロード検証の強化

**問題:**  
`upload_user_image.php` / `import_clone_images_zip.php` はファイルサイズ・MIME タイプを検証しているが、ZIP 内の Path Traversal（`../../` を含むファイル名）の検証が不完全な可能性がある。

**改善案:**
- ZIP 展開時に全エントリのパスを `realpath()` で検証し、展開先ディレクトリ外に出るものを拒否
- ファイル名を UUIDv4 に強制リネームして展開する

---

## 8. パフォーマンス

### 8-1. 🟡 大規模サイトでの解析メモリ不足

**問題:**  
DOM を `DOMDocument` でメモリに展開するため、HTML が数 MB を超えるサイトで `memory_limit` に達する。CSS の取得・パースも同様。

**改善案:**
- `php.ini` の `memory_limit` を 512M 以上に設定
- HTML 取得時に `Content-Length` で事前チェックし、閾値（例: 5MB）超えは警告 + 圧縮解析モード

---

### 8-2. 🟡 CSS 診断スキャン（`scanOutputCssForDiagnostics`）の重さ

**問題:**  
`debug.php` を開くたびに `LpOutputAudit::scanOutputCssForDiagnostics()` が実行される。CSS ファイルが多いワークスペースでは数秒かかる。

**改善案:**
- 診断結果を `data/ws_xxx/css_diagnostics.json` にキャッシュ
- `output/assets/css/` のタイムスタンプと比較し、未変更なら再スキャンしない（`output_unreplaced.json` と同じキャッシュ戦略）

---

### 8-3. 🟢 アセット取得の並列化

**問題:**  
`LpAssetDownloader` のアセット取得はシリアル（順番）実行。100 件のアセットがある場合、1 件平均 200ms だと 20 秒かかる。

**改善案:**
- `curl_multi_*` を使った並列 HTTP 取得（最大 10 並列程度）
- または PHP 8.x の Fiber を使った並列化
- 既取得済み（`asset_map` に存在）のアセットはスキップ（現状の実装を確認・強化）

---

## 9. 保守性・開発体制

### 9-1. 🟡 自動テストなし

**問題:**  
PHPUnit テストが一切存在しない。`LpUrlContext` / `LpGenerator::applyAssetMap` など、バグが起きやすい複雑なロジックがノーテストで稼働している。

**改善案:**
- Composer を導入（ライブラリ管理は最小限にして、PHPUnit のみ）
- `LpUrlContext`・`LpGenerator`・`LpOutputAudit` の単体テストを最優先で追加
- GitHub Actions で push 時に自動テスト実行

---

### 9-2. 🟡 エラーログの構造化なし

**問題:**  
エラー・警告は `error_log()` で Apache ログに出力されるが、ワークスペース ID・ユーザー・URLなどのコンテキストが含まれておらず、障害調査が困難。

**改善案:**
- PSR-3 互換の簡易ロガークラス（`LpLogger`）を作り、構造化ログ（JSON Lines）を `data/app.log` に出力
- ログレベル（DEBUG / INFO / WARNING / ERROR）を `.env` で制御

---

### 9-3. 🟢 `docs/` の整理

**問題:**  
`docs/` に 20+ のマークダウンが存在するが、古い実装メモ（`FIX_*.md`）と現在も有効な仕様書が混在しており、何が最新かわかりにくい。

**改善案:**
- `docs/archive/` ディレクトリに古い FIX メモを移動
- 現行の有効ドキュメント一覧を `docs/INDEX.md` にまとめる

---

### 9-4. 🟢 Composer 不使用の限界

**問題:**  
外部ライブラリを一切使わない方針（Composer 不使用）は依存関係ゼロという利点があるが、以下の車輪の再発明を招いている：
- Google OAuth（`GoogleAuth.php`）
- .env ローダー（`env_load.php`）
- HTTP クライアント（`LpFetcher.php`）
- HTML パーサー（`DOMDocument` のラッパー）

**改善案:**
- 段階的に Composer を導入し、`guzzlehttp/guzzle`（HTTP）・`vlucas/phpdotenv`（.env）など枯れたライブラリを採用
- ただし、現状の動作に問題がない限り急ぐ必要はない

## 10. 将来の機能拡張提案

### 10-1. 🟡 差分更新（元サイトの変更検知）

現状は「解析→編集→生成」の一方向フロー。元の LP が更新された場合に変更箇所を検知し、差分だけ取り込む機能。

### 10-2. 🟡 生成 HTML の品質スコア

`output_unreplaced.json` の件数・未取得アセット数・broken image 数などを点数化し、「生成品質 85/100」のようにダッシュボードに表示。

### 10-3. 🟢 複数言語対応（i18n）

テキスト編集フォームに「言語切り替え」を追加し、日本語版・英語版・中国語版など複数バリアントを同一ワークスペースで管理できるようにする。

### 10-4. 🟢 Webhook / Slack 通知

解析・生成の完了時に Slack や Discord に通知する仕組み。長時間かかる解析を放置しておいても完了を検知できる。

### 10-5. 🟢 テンプレートライブラリ

生成済みの LP を「テンプレート」として保存し、同じ業種・デザインの新規 LP 生成のベースとして再利用できるようにする。
