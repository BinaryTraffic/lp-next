# LP 取り込みとターゲット分析の蓄積・CMS へのフィードバック

*Site Reverse CMS 向け検討メモ（社内・共有用）*  
*見やすい版: [REPORT_LP_INSIGHTS_FEEDBACK.html](./REPORT_LP_INSIGHTS_FEEDBACK.html)（ブラウザ／Word で開きやすい）*

---

## 結論

**取り込んだ LP から「ターゲット・訴求・構造」に関する分析データを蓄積し、集計結果を Site Reverse CMS の分類・診断・提案に返す方針は採用してよい。**

最初はスキーマを軽く決め、`data/` 配下などに **1 LP 1 レコード**で JSON 保存し、十分な件数がたまってから CMS に「ヒント表示」などを足す段階設計が現実的。

---

## 理由（箇条書き）

- **差別化になる**
    - 業種横断で「よくある構成・コピーの型」が貯まる。
    - 単なる HTML 取り込みから、「LP の知見が溜まる基盤」に育てられる。

- **フィードバックの入り口を小さくできる**
    - いきなり CMS 本体に深く組み込まず、別ストア（JSON → 将来的に DB）に分析だけ貯めるところから始められる。

- **事実と解釈を分けられる**
    - DOM から取れる **事実**（見出し数、CTA、フォーム有無など）と、人・LLM が付ける **解釈**（ターゲット仮説）を分離すると、集計・学習がしやすい。

- **既存アーキテクチャと相性がよい**
    - 現状の PHP + JSON のまま `data/insights/` 等のスキーマ付き JSON で足せる。
    - 件数増加後に SQLite 等へ移行する道も自然。

- **リスクを切り分けられる**
    - 参照サイトの利用条件、個人情報・顧客文言の混入に注意し、社内利用・集計のみ・匿名化などポリシーを先に決めれば運用しやすい。

---

## 運用上の補足

- 解析・共有用データには **PII・顧客固有情報の混入**に注意。必要ならハッシュ化・業種タグのみなど設計する。
- 本番 CMS への「返し」は、まず **集計結果の表示**（短文ヒント・業種別の頻出セクションなど）から入れると安全。
